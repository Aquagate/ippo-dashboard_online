## 1. 今回の改善方針

今回の主目的は、新機能追加ではない。以下の4点を優先すること。

1. **現状の棚卸し**  
   何が完成で、何が作業中で、何が壊れているかを見える化する。

2. **再現性の回復**  
   他環境でも起動・テスト・ビルドが同じ手順で再現できるようにする。

3. **文字化け・成果物混在の解消**  
   ファイル管理の衛生状態を整える。

4. **編纂室と羅針盤の完成条件の明文化**  
   どこまでできたら「一区切り」と呼べるかを定義する。

## 4. P0タスク

### P0-01. 作業ツリーの棚卸しと凍結点の作成

#### 目的
現在の変更群を「意味のある束」に分ける。巨大な未整理差分のまま次へ進まない。

#### 実施内容
- 新しい安定化ブランチを切る
- 現在の差分を棚卸しする
- 変更を以下の3区分に分ける
  - A: 完成済みで残すべき変更
  - B: 作業途中だが継続価値のある変更
  - C: 一旦退避または破棄すべき変更
- 棚卸し結果を `docs/audit/` などに残す

#### 推奨コマンド例
```bash
git switch -c stabilize/2026-03-30
mkdir -p docs/audit
git status --short > docs/audit/2026-03-30_worktree_status.txt
git diff --name-only > docs/audit/2026-03-30_changed_files.txt
```

#### 完了条件
- 現在の差分が、A/B/Cの3分類で説明できる
- 「次にコミットしてよい差分」と「まだ触らない差分」が区別されている
- 棚卸しメモがリポジトリ内に残る

---

### P0-02. ファイル名エンコード破損の修復

#### 目的
日本語ファイル名の破損疑いを除去し、将来の検索性・可読性・差分追跡を回復する。

#### 症状
以下のような `#Uxxxx` 形式ファイルが存在する。

- `docs/henzan/#U7d71#U5408#U69cb#U60f3#U66f8_vNext.md`
- `docs/henzan/Phase1_#U5b9f#U88c5#U8a08#U753b#U66f8_#U7de8#U7e82#U5ba4_vNext.md`
- `henzan_vnext_plus_package/01_#U6539#U5584...`

これは本来、少なくとも以下のような名前である可能性が高い。

- `統合構想書_vNext.md`
- `Phase1_実装計画書_編纂室_vNext.md`
- `01_改善指示書_編纂室_AIブリッジ再設計_v0.1.md`

#### 実施内容
- 破損ファイル名の対応表を作る
- `git mv` で正規名へ修正する
- 参照リンクやREADME上の記述も合わせて更新する

#### 完了条件
- `#U` 形式ファイルがゼロになる
- `git status` 上で「削除 + 新規追加」ではなく「rename」として追えるものは追う
- ドキュメント参照切れがない

---

### P0-03. 開発再現性の回復

#### 目的
ZIP起点でも、クリーンな再セットアップ手順で起動・テスト・ビルドできるようにする。

#### 実施内容
- `node_modules/` を成果物から切り離す
- 原則として `npm ci` 再構築前提に揃える
- `README.md` の手順を「ZIP展開後でも再現可能」な形に改修する
- Linux/WSL環境で `.bin` 実行権限が崩れるケースに備える

#### 推奨手順
```bash
rm -rf node_modules dist
npm ci
npm test
npm run build
npm start
```

#### 補足
Linux系環境で `npm test` が `vitest: Permission denied` になる場合は、応急対応として以下を試す。

```bash
chmod +x node_modules/.bin/*
```

ただし本筋は、**ZIPに node_modules を抱え込まないこと** である。

#### 完了条件
- クリーン環境で `npm ci` が通る
- `npm test` が通る
- `npm run build` が通る
- `npm start` で画面が起動する
- READMEの手順だけで再現できる

---

### P0-04. 成果物とソースの境界整理

#### 目的
ソースコードとビルド成果物が混ざる状態を解消する。

#### 現状
`dist/` がGit管理対象に含まれている。

#### 実施内容
- `dist/` をGit管理対象から外すか、管理するなら明確な運用理由を文書化する
- `.gitignore` を整備する
- 配布用ZIPと開発用リポジトリの役割を分ける

#### 推奨方針
通常の開発リポジトリでは **`dist/` は追跡しない**。  
公開配布が必要な場合は、GitHub Releases や配布用ZIPなど別経路に分ける。

#### 完了条件
- `dist/` の扱い方針がREADMEか運用文書に明記されている
- 作業ツリーに不要な成果物差分が増えにくい

---

### P0-05. 編纂室と羅針盤の完成条件を定義する

#### 目的
「進んでいるが終わりが見えない」状態を抜ける。

#### 実施内容
以下の観点で、画面ごとのDoDを定義する。

##### 編纂室
- 入力
- 資産生成
- 資産保存
- 再読み込み
- 編集
- 削除
- AIブリッジ出力検証
- 異常系時の表示

##### 羅針盤
- 候補表示
- 制約評価
- マッチロジック
- 保存
- 再表示
- 空状態表示
- 異常データ時の防御

#### 例: 編纂室DoD
- 新規資産を作成できる
- ページ再読み込み後も残る
- 不正JSONでもアプリ全体が落ちない
- 主要UI操作にトーストまたは画面反応がある
- 最低限のテストがある

#### 完了条件
- `02_DOD.md` か専用文書に、編纂室・羅針盤の受け入れ条件が明記される
- 今後「何をもって完成とするか」が他人にも分かる

## 8. コミット方針

1コミット1目的を徹底すること。

### 例
- `chore(audit): snapshot current worktree and classify changes`
- `fix(docs): restore Japanese filenames in henzan docs`
- `chore(env): remove bundled node_modules and verify clean setup`
- `chore(build): stop tracking dist artifacts`
- `docs(dod): define acceptance criteria for henzan and compass`
- `refactor(future): split futureLab view into smaller modules`

禁止したいのは、

- UI変更
- テスト追加
- ドキュメント修正
- リネーム
- ビルド設定変更

を1コミットに全部混ぜること。

## 12. 用語集

### 作業ツリー
Gitで管理している、いま手元で触っているファイル群のこと。未保存ではなく、未整理の変更も含む。

### 棚卸し
今あるものを全部洗い出して、分類して、何を残すか決めること。倉庫でもコードでも同じで、これをやらないと腐る。

### 再現性
別のPCや後日でも、同じ手順で同じように起動・テスト・ビルドできる性質。

### DoD
Definition of Done の略。何をもって「終わった」と呼ぶかの条件。

### 成果物
ビルドしてできる配布物や生成物。`dist/` が代表例。

### ソース
人が編集する元のコードや文書。基本は `src/` や `docs/` など。

### 回帰テスト
前はできていたことが、あとで壊れていないか確認するテスト。

### リファクタリング
見た目の動きは大きく変えずに、内部構造を整理すること。

### エンコード破損
文字の保存形式が崩れて、ファイル名や文章が化けること。今回の `#Uxxxx` はその疑いが強い。
