# 改善提案メモ: 現状所見と優先順位

## 1. 現状所見
アップロードされたプロジェクトを見る限り、現状は次の状態にある。

### 1-1. すでにあるもの
- `src/ui/views/henzanRoom.ts` があり、編纂室の画面はすでに存在している
- `src/domain/henzan/schema.ts` / `validate.ts` があり、資産スキーマと要確認イベントの型がある
- `src/domain/schema.ts` に `henzanAssets` と `reviewEvents` が DataCache として組み込まれている
- `bootstrap.ts` では `henzan` タブが初期化される
- `docs/henzan/` に vNext / vNext+ の文書が置かれている

### 1-2. まだないもの
- ホビーフォートレスの実装コードが存在しない
- 趣味階層の定義スキーマがない
- 編纂室資産と趣味条件を照合するロジックがない
- `index.html` にホビーフォートレス用タブと画面がない
- DataCache にホビーフォートレスの状態がない

## 2. 判断
この状態なら、次フェーズは **編纂室をさらに深掘るよりも、ホビーフォートレスMVPを最小で立ち上げる** 方が良い。

理由は単純。

1. 編纂室はすでに「動く土台」がある  
2. ホビーフォートレスはまだゼロなので、価値の伸びしろが大きい  
3. 編纂室の出力が本当に使えるかを検証するには、下流であるホビーフォートレスを作るのが早い  
4. 夢だけのページではなく、編纂室資産が実際に次の階層判定へ使われるかを見たい  

## 3. 次フェーズの優先順位
### 最優先
**Phase 2A: ホビーフォートレスMVP**
- 1趣味だけ
- 3階層だけ
- 条件照合だけ
- ミッション提示だけ

### 後回し
- 多趣味対応
- AI自動で階層定義を作る
- 複雑なグラフ
- 重いゲーム化
- 編纂室側の過剰自動化

## 4. まずやるべき具体
### P0
- 趣味階層スキーマを作る
- 釣りなど 1 趣味のサンプル定義を置く
- 編纂室資産との照合関数を作る
- ホビーフォートレス画面を 1 ページ追加する

### P1
- 「次の階層に進む条件を満たしています」を表示する
- 不足条件を出す
- 小さなミッションを 1〜3 件出す

### P2
- 制約資源（時間・体力・お金・家族）を表示に反映する
- 複数趣味対応
- AI補助によるミッション生成

## 5. 今回の設計思想
ホビーフォートレスは「義務化装置」ではなく、**次階層を覗く装置**として実装する。
ここをミスると、趣味ページがただの圧力装置になる。人間は趣味にまでKPIを持ち込んで壊すので、そこはちゃんと避ける。

## 6. 結論
次に作るべきは、**改善提案書より、ホビーフォートレスMVPの実装指示書**。
理由は、現状のコードがすでに編纂室基盤を持っているからである。
