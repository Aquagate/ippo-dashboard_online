# FUTURELAB_LEAP_UPGRADE.md
うぜん向け：FutureLabの **Leap強化** と **スマホ表示最適化（PC据え置き）** の実装指示書。  
IDE/エージェントにそのまま渡して実装できるよう、タスクを分割し、DoD（完了条件）を付与しています。

---

## まずここを直せば勝てる（3行）
1. **過去ログの圧縮（evidence_summary）** をアプリ側で生成し、「踏襲」を安定供給する。  
2. Leap出力を **3セクション固定（Evidence Mirror / Chaos Leap / Action Seeds）** にして、飛躍を制御する。  
3. FutureLab表示を **モバイル幅だけ** 読みやすくし、長文をセクション単位で折り畳めるようにする。  

---

## 重要ルール（安全・品質）
- 捏造禁止：過去ログの“事実”は `evidence_summary` のみを根拠として扱う
- 占い化禁止：未来は断定しない（「選択肢の提示」に徹する）
- 医療診断禁止：体調や病名の断定はしない
- 秘密情報を出さない：ログや画面にトークン/設定値の詳細を増やさない
- PC版は現状維持：スマホ最適化は FutureLabの表示範囲に閉じる

---

## ASSUMPTIONS（仮定）
- FutureLabは「プロンプトテンプレ + 入力データ（ログ等） + モデル呼び出し」で生成している
- Leap/Baseline/Guardrail は同一箇所で分岐している（例：`src/ui/views/futureLab.ts` 周辺）
- 生成結果はMarkdownまたはプレーンテキストで表示されている
- 既存のAPI呼び出し（モデル）は維持できる（ここでは“プロンプト/表示/入力整形”に集中）

# バックログ（P1想定）

## P1-05
- ID: P1-05
- タイトル: Leap三段プロンプト
- 目的: Leapが「いつもの延長」になってしまう問題を潰し、踏襲しつつ“カオス超進化”を安定して出す
- 変更内容:
  - （推奨）`src/domain/future/prompt.ts`（新規）: FutureLab用プロンプト生成ロジックの切り出し
  - `src/ui/views/futureLab.ts`（既存）: Leap生成処理を「3セクション固定」に変更
- 手順:
  1. Leap出力を **必ず3セクション構成**にする（順序固定）
     - `## Evidence Mirror`
     - `## Chaos Leap`
     - `## Action Seeds`
  2. “カオス”は **発想** に限定し、事実根拠は `evidence_summary` に固定する
  3. 出力はスマホで読みやすいように、段落短め＋箇条書き多めをルール化する
- DoD:
  - Leapの生成結果が必ず上記3見出しを含む
  - 少なくとも3回連続生成して、毎回「踏襲（Evidence）」と「飛躍（Chaos）」の両方が含まれる
  - Baseline/Guardrail の挙動・出力に影響がない
- 影響範囲: FutureLab生成（Leapのみ）、表示（見出し分割する場合）
- リスク/注意:
  - 3回APIコールにするとコスト/時間が増えるため、まずは **1回の生成で3セクション出力** を推奨
- 推定工数感: M

### Leapプロンプト（テンプレ：変数名は実装に合わせて置換）
```text
# ROLE
あなたは「未来シミュレーションエンジン：Leap担当」です。
目的は当てることではなく、ユーザーの選択肢を増やし、インスピレーションを生むことです。

# INPUTS
- 現在日付: {{today}}
- ユーザーの過去ログ要約（事実）: {{evidence_summary}}
- 直近の一歩（事実）: {{recent_ippo_list}}
- 制約: 占い化禁止 / 医療診断禁止 / 根拠のない断定禁止 / 個人情報を出さない

# OUTPUT RULES
- 出力はスマホで読みやすく。1段落は短く、箇条書きを多用。
- 必ず下記3セクション構成。順序を変えない。
- 根拠は「過去ログ要約」から引用・言い換えで示す（捏造しない）。

## Evidence Mirror
- 過去ログ要約から読み取れる「繰り返し」「強み」「避けるべき罠」を5〜8項目で列挙。
- “踏襲する軸”を3つに圧縮して宣言する。

## Chaos Leap
- 踏襲する軸を壊さずに、あえて矛盾を注入して10Xの飛躍案を5本出す。
- ただし各案は「現実に接続する導線（何が変われば可能か）」も1行添える。
- うち2本は“異世界”レベルで飛ぶ（例: 役割/環境/時間の使い方を反転させる）。

## Action Seeds
- 上の飛躍案から「今日できる1分」3つ、「今週できる30分」3つを提案する。
- それぞれに「成功条件（完了の判定）」を1行で書く。
```

## P1-06
- ID: P1-06
- タイトル: evidence要約生成
- 目的: “過去の一歩踏襲”をモデル任せにせず、アプリ側で安定供給する
- 変更内容:
  - `src/domain/future/evidence.ts`（新規）: 過去ログから `evidence_summary` を生成
- 手順:
  1. 過去N日（推奨: 30日 or 60日）のIppoを取得
  2. 次を抽出して箇条書きでまとめる（文字数上限あり）
     - カテゴリ頻度（上位/下位）
     - 時間帯の偏り（朝/昼/夕/夜）
     - 連続性（継続していること / 途切れやすいこと）
     - （あれば）抵抗感/難易度の傾向
     - “避けるべき罠”（例: 夜に崩れる、入力が重いと続かない等）
  3. 文字数上限（推奨: 1200〜2000文字）を超えないように切り詰める
  4. ログ本文はそのまま大量投入しない（秘密情報混入対策）
- DoD:
  - `evidence_summary` が毎回一定の構造（見出し＋箇条書き）で生成される
  - ログが少ない/ゼロでも破綻しない（「データ不足」扱いで出力）
  - Leapプロンプトへ `evidence_summary` を渡せる
- 影響範囲: FutureLab入力整形
- リスク/注意: ここで生成するのは「要約」であって「評価」や「断定」ではない
- 推定工数感: M

## P1-07
- ID: P1-07
- タイトル: FutureLabモバイル整形
- 目的: スマホで未来ラボが読みやすくなるように（PCは現状維持）
- 変更内容:
  - FutureLab表示部分に限定してCSS/DOM構造を調整
  - セクションごとに折り畳み（`details/summary` 等）を導入（軽量）
- 手順:
  1. FutureLab出力を見出しで分割し、セクション単位のDOMにする
     - `Evidence Mirror / Chaos Leap / Action Seeds` が分割単位

     - フォントサイズ/行間/余白を増やす
     - セクション折り畳みを有効化
     - “上へ戻る”導線を小さく固定（任意）
  3. PC幅では見た目がほぼ変わらないことを確認
- DoD:
  - 390px幅（iPhone想定）で見出し・箇条書きが潰れない
  - セクション折り畳みが動作する
  - PC幅で既存レイアウトが大きく崩れない
- 影響範囲: `src/ui/views/futureLab.ts` と FutureLab表示用CSS（または inline style）
- リスク/注意: 全体CSSへ影響を広げない（FutureLabルートのclass配下のみ）
- 推定工数感: M
